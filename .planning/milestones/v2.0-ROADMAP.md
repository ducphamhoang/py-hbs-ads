## Milestone v1.0 - Python rewrite foundation

### Milestone Definition of Done

This milestone is done when:

1. The Python repo exposes the documented `hbs-ads` command tree through a coherent CLI package.
2. Foundations for config, workspace resolution, errors, logging, subprocess execution, and SQLite are in place.
3. Critical workflow areas are migrated behind feature-first Python services rather than ad hoc scripts.
4. The rewrite is tested and can exercise representative legacy-style workflows without relying on the Go codebase.

### Phase 1: Project Bootstrap and CLI Skeleton

**Goal:** Create a real Python application skeleton that proves the feature-first architecture, exposes the command tree, and establishes typed app boundaries for future phases.

**Requirements:** FOUND-01, FOUND-02, FOUND-03, FOUND-04

**Success Criteria**:
1. `pyproject.toml` defines a working package and CLI entrypoint.
2. `src/hbs_ads` contains `cli`, `app`, `core`, `infra`, and `features` packages.
3. The CLI exposes the documented command groups with placeholder-safe command handlers.
4. Foundation types exist for settings, errors, and output rendering.
5. Initial tests prove the CLI boots and help output is reachable.

### Phase 2: Config, Workspace, and Persistence Foundations

**Goal:** Implement the shared runtime concerns that every feature depends on before deeper workflow migration begins.

**Depends on:** Phase 1

**Requirements:** FOUND-05, BOOT-01, BOOT-02, BOOT-03, BOOT-04

**Success Criteria**:
1. Config precedence across file, env, and CLI flags is implemented and tested.
2. Workspace initialization and path resolution are explicit and reusable.
3. SQLite bootstrap and migrations are wired through the application layer.
4. Shared subprocess execution boundaries support dry-run and structured results.

### Phase 3: Ingest, Trim, and Tagging Migration

**Goal:** Migrate the first core operator workflows into Python feature services with explicit request/result models and safe side-effect boundaries.

**Depends on:** Phase 2

**Requirements:** INGEST-01, TRIM-01, TAG-01

**Success Criteria**:
1. Ingest, trim, and tag commands execute through feature services, not thin wrappers around shell glue.
2. Dry-run behavior is explicit and tested for mutating operations.
3. Shared workspace, config, and subprocess foundations are reused consistently.

### Phase 4: Variants, Hooks, and Pipeline Orchestration

**Goal:** Implement the central creative assembly flows and the orchestration path that connects them.

**Depends on:** Phase 3

**Requirements:** VAR-01, HOOK-01, PIPE-01

**Success Criteria**:
1. Variant generation, assembly, export, validation, and archive paths exist behind feature services.
2. Hook assembly integrates cleanly with the variant workflow.
3. Pipeline orchestration coordinates features without collapsing boundaries.

### Phase 5: External Integrations and Reporting

**Goal:** Add the remaining integrations and reporting workflows through explicit adapter seams and feature-owned behavior.

**Depends on:** Phase 4

**Requirements:** SHARE-01, COMP-01, PERF-01, NOTIFY-01, VOICE-01

**Success Criteria**:
1. SharePoint, notify, competitor, perf, and voiceover commands exist through the Python CLI.
2. External dependencies are isolated behind clear adapter boundaries.
3. Integration behavior is covered by tests or fixture-driven verification where live systems are unavailable.

### Phase 6: Hardening, Parity Verification, and Cutover Readiness

**Goal:** Prove the rewrite is operationally credible and document how to cut over phase by phase from the current repo.

**Depends on:** Phase 5

**Requirements:** QUAL-01, QUAL-02, QUAL-03, MIG-01

**Success Criteria**:
1. Core workflow behavior is covered by unit and integration tests.
2. Representative legacy-style workflows run successfully against the Python CLI.
3. Operator and developer docs explain how to install, verify, and migrate.
4. Remaining parity gaps are explicit and tracked, not hidden.

## Milestone v2.0 - Hermes Discord ↔ Notion Scrum Level 2+3

### Milestone Definition of Done

This milestone is done when:

1. Shared workflow modules (`models.py`, `person_resolution.py`, `prompt_store.py`, `audit.py`) replace all duplicated logic from existing Level 1 scripts.
2. Three Level 3 operator entrypoints run the full inbound pipeline, pending prompt creation, and preflight validation as single commands with stable JSON result envelopes.
3. Notion-specific code is isolated behind an adapter boundary so the generic pattern can be reused with a different backend.
4. Pattern-level documentation describes the reusable "shared-thread attributed automation" design, and all shared modules and entrypoints are covered by tests.

- [x] **Phase 10: Pattern Documentation and Test Coverage** — Write the pattern docs and close test coverage over all shared modules and entrypoint pipelines.

### Phase 8: Shared Workflow Modules (Level 2)

**Goal:** Operators and scripts share a single, canonical implementation of workflow models, person resolution, prompt lifecycle, and audit so no logic is duplicated across scripts.

**Depends on:** Phase 7 (v1.0 complete)

**Requirements:** MOD-01, MOD-02, MOD-03, MOD-04, MOD-05

**Success Criteria**:
1. A single import of `models.py` provides typed `InboundEvent`, `PromptRecord`, `MatchResult`, and `UpdatePlan` objects usable by any script.
2. `person_resolution.py` is the only place that does platform identity lookup, canonical-person resolution, pending-person surfacing, and actor label generation.
3. `prompt_store.py` owns the full prompt lifecycle — load, save, append, retrieve open prompts, and mark state transitions — with no duplicate implementations elsewhere.
4. `audit.py` handles all audit event formatting and append-only writes; no script formats or writes audit records independently.
5. `resolve_person.py`, `lookup_notion_person.py`, and `record_pending_prompt.py` are verified thin wrappers with zero duplicated logic.

### Phase 9: Operator Entrypoints, Contracts, and Adapter Boundary

**Goal:** Operators can run the full inbound pipeline, create pending prompts, and run preflight validation as single commands, with every write-capable command returning a stable JSON envelope and Notion-specific code isolated behind a clear adapter boundary.

**Depends on:** Phase 8

**Requirements:** ENTRY-01, ENTRY-02, ENTRY-03, ENTRY-04, CONTRACT-01, CONTRACT-02, ADAPTER-01

**Success Criteria**:
1. Running `process_inbound_reply.py` executes resolve → match → plan → apply in one command and exits with a valid JSON result envelope.
2. Running `create_pending_prompt.py` records a validated pending workflow object with an audit write and exits with a valid JSON result envelope.
3. Running `preflight.py` validates registry integrity, prompt integrity, unresolved people, duplicate IDs, and state consistency in one command.
4. Every write-capable entrypoint defaults to dry-run and requires an explicit flag to execute; the JSON output shape is identical in both modes; `prompt.answered` is never set in dry-run.
5. Notion-specific logic (user-ID mapping, page/comment patch calls, task semantics) lives entirely in an adapter module and is not referenced by the generic pipeline code.

**Plans:** 4 plans

Plans:
- [x] 09-01-PLAN.md — Create stable result contracts and Notion adapter boundary
- [x] 09-02-PLAN.md — Add create_pending_prompt.py with validation, audit, and envelope output
- [x] 09-03-PLAN.md — Add process_inbound_reply.py end-to-end dry-run/execute pipeline
- [x] 09-04-PLAN.md — Add preflight.py operational state health check

### Phase 10: Pattern Documentation and Test Coverage

**Goal:** The "shared-thread attributed automation" pattern is documented well enough for a future adopter to apply it to a new backend, and all shared modules and entrypoint pipelines are covered by automated tests.

**Depends on:** Phase 9

**Requirements:** DOCS-01, TEST-01, TEST-02

**Success Criteria**:
1. Documentation identifies what is generic (sender resolution, prompt lifecycle, reply matching, audit discipline) versus what is Notion-specific, and describes how a new workflow adopts the pattern.
2. Tests cover all four shared modules including edge cases and schema validation for `models.py`, state-transition rules in `prompt_store.py`, resolution fallbacks in `person_resolution.py`, and event-type conventions in `audit.py`.
3. Tests exercise all three entrypoint pipelines (`process_inbound_reply`, `create_pending_prompt`, `preflight`) in both dry-run and execute modes and assert identical output shapes.
